# 개발 일지: 도서 기록 시스템 GitHub DB 연동 및 안정화 (2026-04-09)

오늘의 주 작업은 로컬 저장 방식에 의존하던 도서 기록 시스템을 **GitHub API 기반의 클라우드 저장소(DB)**로 전환하고, 이를 Vercel 배포 환경에서 완벽하게 작동하도록 안정화하는 것이었습니다.

## 1. 주요 작업 내용

### 🛠️ GitHub 연동 고도화 및 보안 강화
- **환경 변수 분리**: 하드코딩되어 있던 GitHub 저장소 정보(`OWNER`, `REPO`)를 환경 변수(`GITHUB_OWNER`, `GITHUB_REPO`)로 전환하여 유연성과 보안을 확보했습니다.
- **연동 유효성 검증**: 서버 시작 시 GitHub 토큰과 저장소 권한을 사전에 확인하는 로직을 추가하여, 연동 실패 시 사용자에게 명확한 가이드를 제공하고 로컬 파일 모드로 안전하게 전환(Fallback)되도록 했습니다.

### 🧱 Vercel 배포 환경 최적화 (구조 개선)
- **백엔드 구조 재편**: Vercel의 서버리스 환경에서 Express 서버가 안정적으로 인식되도록 `server.js`를 `api/server.js`로 이동하고 `vercel.json` 라우팅 설정을 업데이트했습니다.
- **레이스 컨디션(Race Condition) 해결**: 서버 부팅 시 GitHub 데이터를 다 불러오기 전에 프론트엔드 요청에 응답하여 빈 화면이 뜨던 문제를 `initializationPromise` 도입을 통해 해결했습니다.

### 🛡️ 데이터 보호 및 동기화
- **안전한 마이그레이션**: 브라우저 로컬 데이터를 클라우드로 옮기는 과정에서 서버 저장 성공 여부를 확인한 후 로컬 데이터를 삭제하도록 로직을 보강하여 데이터 유실 가능성을 차단했습니다.
- **데이터 복구 및 업로드**: 로컬 환경에만 남아있던 5권의 도서 데이터를 스크립트를 통해 GitHub 저장소로 수동 동기화하여 Vercel에서도 즉시 보이도록 조치했습니다.

### ♿ 웹 표준 및 접근성 개선
- **폼 어트리뷰트 보강**: 도서 등록 폼의 모든 필드에 `id`와 `name` 속성을 추가하고 레이블을 연결하여 접근성을 높이고 브라우저의 자동 완성 기능을 지원하게 되었습니다.

## 2. 해결한 기술적 난제
- **API 응답 형식 오류 (n.filter is not function)**: 서버 초기화 중 또는 데이터 파싱 시 오류가 발생했을 때 프론트엔드가 비정상적인 데이터를 받지 않도록 백엔드와 프론트엔드 양측에 **방어적 프로그래밍(Defensive Programming)**을 적용했습니다.
- **Vercel 라우팅 이슈**: API 요청이 프론트엔드 HTML로 리다이렉트되던 현상을 Vercel 표준 디렉토리 구조(`api/`) 적용을 통해 근본적으로 해결했습니다.
